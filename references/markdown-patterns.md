# Markdown Patterns for rabbitQ-skill-lark-xhs

这份约定用于准备飞书导出的 Markdown。转换器直接解析 Markdown 与附件，无需其他中间格式。

## 封面标题与副标题

封面大标题、副标题与正文标题**分开识别**。

支持三种写法（优先级从左到右）：

1. **CLI**：`--title` / `--subtitle`
2. **YAML frontmatter**：`title:` / `subtitle:`
3. **文首中文标签**（正文最前面，中间可空行）：
   - `标题：…` / `大标题：…` / `封面标题：…`
   - `副标题：…` / `封面副标题：…`
   - 中英文冒号都行（`：` / `:`）

识别后会**从正文删掉**这些标签行，只出现在封面。缺副标题时，才由执行转换的 AI 读全文补一句。

**何时跳过正文里的某个 `#` 标题？**

- 仅当该行标题**文案与已解析的封面大标题相同**时跳过，避免封面与正文重复。
- **不会**一律把文档第一个 `#` 当封面并全部跳过——只有在你**没有**用 frontmatter / CLI 指定大标题、且首个 `#` 被采纳为封面时，同文标题才不进正文。
- 若你用 frontmatter 指定了与飞书导出 `#` **不同**的大标题，正文里的 `#` 会按正常深度映射（这是预期行为）。

## 正文标题（一级 / 二级）

在排除「与封面大标题同文」的标题之后，正文标题按**文内实际 `#` 深度**动态映射：

- 最浅的那种 `#` → **一级标题**（典型飞书稿：`## 01 它到底做什么` → 大号 01/02 + 通栏线）
- 第二浅的那种 `#` → **二级标题**（如 `##### 快速开始` → 强调色 + 下划线，无编号）
- 更深的 `#` → 加粗正文段落

**两种常见稿型：**

| 稿型 | 封面大标题来源 | 正文首个 `#` | `## 01 …` |
|---|---|---|---|
| 飞书默认导出（无 frontmatter） | 首个 `#` | 与封面同文 → 跳过 | 正文一级 |
| 指定 frontmatter 大标题 | frontmatter | 与封面不同 → **导语加粗**（有 `##` 时） | 正文一级 |

例：典型飞书稿 `# 文档名` + `## 01 …` + `##### 小节` 时，`##` 是一级、`#####` 是二级。

```markdown
---
title: "AI 视频一眼假怎么破？"
subtitle: "这次不是讲概念，是一次真实踩坑复盘。"
---
```

若省略 `subtitle`，交给 AI 在转换前读文档生成，例如：

```bash
node scripts/convert.js article.md --subtitle "一眼分清假视频问题，附可复用提示词思路"
```

## 标题

- **封面大标题**：frontmatter `title` / CLI `--title` /（未显式指定时）文档第一个 `#` / 文件名。
- **封面副标题**：frontmatter `subtitle` / CLI `--subtitle` / AI 总结。
- 正文里**仅与封面大标题同文**的 `#` 标题会跳过；其余 `#` / `##` 按深度映射：
  - 最浅 `#` → **一级标题**（自动编号 01/02 + 通栏分隔线）
  - 第二浅 `#` → **二级标题**（强调色字 + 2px 浅色下划线，无编号）
  - 更深的 `#` → 加粗正文段落
- 例（飞书默认）：`# 文档名` + `## 01 …` + `##### 小节` → `##` 是一级、`#####` 是二级。
- 例（指定 frontmatter）：封面用 frontmatter，正文 `#` 与封面不同则保留为正文标题。
- 已写 `01`、`02` 的一级标题会保留编号；没有编号时自动连续编号。
- 编号和标题属于同一个块，不能通过空行拆散。

## 正文与行内样式

- 普通段落会成为可按行跨页的正文。
- Markdown 里的**空行**仅在「正文段落 ↔ 正文段落」或「卡片 → 正文段落」时保留为可见间距（`xhs-manual-blank`）。
- 飞书导出在图片、标题、列表、卡片周围常见的空行**不会**自动转成空行，避免块与块之间被莫名撑开。
- `**加粗**` 保留为粗体。
- 整段加粗且长度足够时，可自动推断为卡片。
- 只有明确以「高亮 / 划重点 / 卡片 / 注意 / 结论 / 金句 / 关键 / 判断 / 提醒 / 重点」等标签开头的部分加粗段落，才按卡片处理。

## 引用

```markdown
> 引用块适合放观点、说明和补充背景。
```

飞书原生引用导出后通常也是 `>`。引用结构优先于关键词，即使内容出现“金句”也不会变成卡片。

## 列表

```markdown
- 无序项目 A
- 无序项目 B

1. 有序项目 A
2. 有序项目 B
```

列表结构优先于内容关键词。列表项中出现“卡片”“重点”等词，仍保持列表。

Studio 内在列表项中按 Enter 会拆出同类型的新列表项：有序列表保持有序、无序列表保持圆点，粗体和光标后的正文必须完整保留。

## 表格

推荐使用标准 GFM pipe table：

```markdown
| 模式 | 适合 | 页数 |
| --- | --- | ---: |
| 有封面图 | 视觉强、产品感 | 6 |
| 无封面图 | 干货长文、教程 | 8 |
```

- 转换器通过“表头行 + `---` 分隔行”识别表格，不会把普通竖线文本误判为表格。
- 支持对齐冒号、首尾省略 `|`，以及单元格内用 `\|` 表示竖线。
- 飞书导出为 HTML `<table>` 时也会保留行列。
- 建议控制在 2–5 列；列太多应先精简，不要依赖横向滚动。

## 图片

```markdown
![图片说明](图片和附件/image%201.png)
```

- 推荐把 Markdown 和附件目录一起传入转换器。
- 支持中文、空格和 URL 编码路径。
- 连续竖图可能自动并排。

## 视频

视频不会进入小红书图文：

```markdown
演示视频.mov
```

转换器会跳过视频并生成提示。需要视频内容时，先截帧为图片，或在小红书发布页单独上传。

## 手动分页

单独一行的 `---`（也支持 `***`、`___`）表示**强制换页**。下一行内容会从新一页顶部开始；分页符本身不会出现在导出图里。

```markdown
## 03 这一页必须在这里结束

最后一段正文。

---

## 04 从这里起是新一页
```

注意：表格分隔行 `| --- | --- |` 不会被当成手动分页。不要用大量空行凑页码；高度仍由 Studio 按 3:4 版心自动测量，手动分页只是在自动分页之上加「硬断点」。
